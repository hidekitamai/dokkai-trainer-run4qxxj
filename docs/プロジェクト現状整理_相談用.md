# 読解トレーニング — プロジェクト現状整理（コンサルティング相談用）

> 作成日: 2026-06-14
> 目的: AIコンサルタントに現状の構造を説明し、より良い構造へのアドバイスを得るための資料
> 想定読者: 技術コンサルタント（このリポジトリを初めて見る前提で書いています）

---

## 0. 30秒サマリー（最初に読む欄）

- 小学生2人（兄・妹）向けの **Web版・読解クイズ** を、ほぼ毎日1日分ずつ作って公開している個人プロジェクトです。
- 制作には **Claude Code（AIコーディングツール）** を使っています。作者自身はエンジニアではありません。
- いまの作り方は「**1日分 = 完成済みHTMLファイル2枚**」で、各ファイルが見た目・採点プログラムまで丸ごと自己完結しています。
- そのため **共通部分が全ファイルにコピーされて重複**しており、「直すときに全ファイルを直す必要がある」のが目下の悩みです。
- 相談したいこと: **(A) ファイル/フォルダ構成の改善** と **(B) Claude Code をうまく使うための準備（CLAUDE.md 等）**。

## 1. このプロジェクトは何か

| 項目 | 内容 |
|------|------|
| 名称 | 読解トレーニング（Dokkai Trainer） |
| 目的 | 小学生の子ども2人に、毎日1回ぶんの読解（文章読み取り）クイズを解いてもらう |
| 利用者 | 子ども2人（呼称: **兄 = sho** / **妹 = rin**） |
| 形態 | ブラウザで開くWebページ。スマホ・PCで動く |
| 公開方法 | GitHub Pages（推定）で限定的に公開。検索除けあり（`noindex`）、URLを知っている人だけアクセスする想定 |
| 結果記録 | 解答後、点数を **Googleフォーム** に自動送信＋端末内にも保存 |
| 制作ツール | Claude Code（AIにコードを書いてもらうツール） |

### 1日分の中身（子どもが体験する流れ）
1. 本文（短い文章）が表示される
2. 選択式の設問が5問ほど出る
3. 1問ずつ ○×と正解が表示される
4. 全問終わると合計点と「がんばったね」等のメッセージが出る
5. 点数がGoogleフォームと端末に記録される

兄（sho）は青テーマ・難易度高め、妹（rin）はピンクテーマ・やさしめ、というように **同じ仕組みで難易度と見た目だけ変えて** 2人分を用意しています。

## 3. 1ファイルの中身の構造（重要）

各HTMLファイルは、3つの部品が **1枚に全部詰め込まれた** 自己完結型です。

| 部品 | 役割 | ファイル間での違い |
|------|------|--------------------|
| **HTMLの骨組み** | 画面の枠組み | ほぼ共通 |
| **CSS（見た目・デザイン）** | 色、レイアウト、ボタンの形 | **完全に共通**（テーマ色だけ兄=青/妹=ピンク） |
| **JavaScript（採点プログラム）** | 採点・点数集計・Googleフォーム送信 | **全ファイル完全に同一** |
| **問題データ（JSON）** | 本文・設問・選択肢・正解 | ★ここだけが日替わりで違う |

### 実測で分かった「共通部分の重複」

全ファイルを問題データ部分だけ伏せて比較したところ、**CSS と JavaScript（採点プログラム）は全ファイルで完全に一致**していました。
日ごとに違うのは実質、次の4点だけです:

1. 問題データ（本文・設問のJSON）
2. 日番号（タイトルと見出しの数字）
3. レベル表示（Lv1 / Lv2 など）
4. 人物（兄/妹 — 名称テキストとテーマ色）

> つまり **約80%のコード（採点プログラムとデザイン）が、42ファイルすべてに同じ内容でコピーされている** 状態です。

## 4. 現状の作り方（Claude Code の使い方）

> ※この節は、コミット履歴とファイルの状態からの**推定**を含みます。実際の運用と違う点があれば修正してください。

- ほぼ毎日、Claude Code に依頼して **その日の2ファイル（兄・妹）を生成**している。
- 生成のたびに、デザインと採点プログラムを含む **HTML一式まるごと** を作り直している（＝共通部分も毎回再生成）。
- 出来たファイルをコミットして GitHub に push、GitHub Pages で公開。
- `CLAUDE.md`（Claudeへの指示書）などの設定ファイルは **現状なし**。毎回ゼロから文脈を説明している。

## 5. 現状の課題（相談したいポイント）

### 課題A: 共通部分の重複（最優先）
- デザインや採点ロジックを1か所変えたいだけでも、**42ファイル全部を直す**必要がある。
- 手作業・AI生成のたびに **ファイル間で微妙なズレ（バグ）** が生まれるリスク。
- 毎回フルセットを生成するため、**AIへの指示・生成が冗長**でコストもかさむ。

### 課題B: フォルダ構成
- すべてが直下に平置きで、今後日数が増えると **一覧性が下がる**。
- ただし公開URLを変えたくない事情があり、**安易なフォルダ移動はリンク切れの懸念**。

### 課題C: Claude Code の使い方の土台づくり
- プロジェクトの前提（命名規則・データ形式・作業手順）を **AIが毎回引き継げる仕組み** がない。
- 「どう設定ファイルを整備すれば、毎日の依頼がラクで安定するか」を知りたい。

## 6. 作者が現時点で検討している改善案（たたき台）

コンサルタントへの相談前に、AIから受けた助言をもとに考えている方向性です。**これが妥当か、もっと良い方法はないか** を相談したい。

### 案A: 共通部分を1か所にまとめる（重複の解消）
- 全ファイル共通の **CSS と 採点プログラム** を、それぞれ独立した1ファイルに切り出す。
- 各日ファイルは「問題データ ＋ 共通ファイルの読み込み」だけの薄い中身にする。
- ねらい: **直すのは1か所だけ／毎日の作業は問題データを書くだけ** にする。

```
変更後イメージ（公開URLは変えない方針）
├── index.html
├── assets/                 ← 新設：共通部品の置き場
│   ├── style.css           ← 共通デザイン（1枚）
│   └── quiz.js             ← 共通の採点プログラム（1枚）
├── day46_sho.html          ← 中身は「問題データ＋読み込み指定」だけに軽量化
├── day46_rin.html
└── CLAUDE.md               ← 新設：Claudeへの指示書
```

### 案B: さらに進めて「データとひな形の分離」（将来的な選択肢）
- 問題を `data/day46_sho.json` のようにデータだけで持ち、ひな形1枚＋生成スクリプトでHTMLを自動生成する。
- メリットは大きいが「生成の手間」が増えるため、**今すぐは案Aで十分** という認識。

### 案C: CLAUDE.md（Claudeへの指示書）を1枚用意
- プロジェクトの前提・命名規則・データ形式・作業手順を1枚にまとめ、AIが毎回自動で読む状態にする。
- 規模的に **1枚で十分**（分割は不要）という認識。

## 7. コンサルタントに聞きたいこと（質問リスト案）

> 当日そのまま使える形で質問を用意しました。優先度順。

1. **構成の方向性**: 案A（共通部分の切り出し）はこのプロジェクト規模で妥当か。もっと適した構成はあるか。
2. **ビルドの要否**: 案B（データ＋自動生成）に進む価値はあるか。非エンジニアでも維持できる範囲か。判断基準は？
3. **公開URLの扱い**: フォルダ整理とリンク維持を両立する良い方法は？（GitHub Pages の前提で）
4. **Claude Code の設計**: `CLAUDE.md` に何を書くと毎日の作業が安定・効率化するか。推奨テンプレートはあるか。
5. **品質の担保**: AI生成物のばらつき・ミスを防ぐ仕組み（テスト・チェックリスト・サンプル提示など）の作り方。
6. **運用の自動化**: 「毎日1日分追加」を、もっと省力化・自動化する現実的な手段はあるか。
7. **データの活用**: Googleフォームに溜まる解答結果を、今後どう活かせるか（進捗の可視化など）。
8. **将来の拡張**: 利用者や教科を増やす場合、いまのうちに備えるべき構造上のポイントは？

## 8. 用語ミニ解説（必要に応じて）

| 用語 | かんたんな意味 |
|------|----------------|
| HTML | Webページの中身（文字・問題そのもの） |
| CSS | 見た目（色・レイアウト・デザイン） |
| JavaScript | 動き（採点する・点数を出す） |
| GitHub Pages | GitHub上のファイルをそのままWeb公開する無料の仕組み |
| CLAUDE.md | Claude Code が会話の最初に自動で読む「プロジェクトの説明メモ」 |
| ビルド/生成スクリプト | 元データから完成HTMLを自動で作る小さなプログラム |
| 重複（重複コード） | 同じ内容が複数ファイルにコピーされている状態。直すとき全部直す必要がありミスの元 |
