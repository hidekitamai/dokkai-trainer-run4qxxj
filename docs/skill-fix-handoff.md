# 書面チェックスキル修正の引き継ぎ書（パソコンのClaude Code用）

作成日：２０２６年７月１２日
作業ブランチ：`claude/existing-skill-fixes-7cr8mr`（リポジトリ：hidekitamai/dokkai-trainer-run4qxxj）

## １　決定事項（このセッションで確定）

1. **採用**：依頼者目線チェック（観点４）を新設し、既存スキル「/bengo-rexiew」に追加する。
   - 背景：書面を依頼者に送付して確認を求める運用が今後頻発するため、法律の素人である依頼者からの指摘・疑問に耐えられる書面にする。
2. **見送り**：２つの書面チェックスキル（内容チェックと誤字・矛盾チェック）の統合は行わない。現行の２スキル体制を維持する。
3. **採用**：スキルマップ上の並び順を「/bengo-rexiew」→「goji-check」の順に変更する（現状はgoji-checkが先）。
   - 趣旨：内容チェック（bengo-rexiew）を先に行い、内容修正後に誤字・矛盾チェック（goji-check）を行う運用順序に合わせる。

※スキル名の表記について：本書では「bengo-rexiew」と記載しているが、ローカルの実際のフォルダ名（bengo-reviewの可能性あり）を確認し、実名に合わせて作業すること。

## ２　パソコン側での作業手順

1. ローカルのスキル保存場所（`~/.claude/skills/`または各プロジェクトの`.claude/skills/`）から「bengo-rexiew」のSKILL.mdを開く。
2. 既存の３つの観点の後に、下記「３」の観点４を追加する。既存の３観点の記載には手を加えない。
3. 観点４の追加に伴い、SKILL.md冒頭のdescriptionや観点数への言及（「３つの観点」等の文言）があれば「４つの観点」に修正する。
4. スキルマップ（スキル一覧を記載したファイル。CLAUDE.md、README、skill-map等の名称の可能性あり）を探し、「goji-check」と「bengo-rexiew」の並び順を入れ替えて「bengo-rexiew」→「goji-check」の順にする。
5. 実際の書面１通で「/bengo-rexiew」をテスト実行し、観点４の指摘が出力されることを確認する。

## ３　追加する観点４の本文（そのまま貼り付け用）

既存スキルの見出しの階層・体裁に合わせて調整すること。

```markdown
### 観点４：依頼者目線チェック

法律の素人である依頼者がこの書面を読むことを前提に、以下を確認する。

1. **わかりやすさ**
   - 専門用語・法律用語で、依頼者が意味を取れないものはないか（必要なら注記または言い換え候補を提案する）
   - 一文が長すぎて趣旨が追えない箇所はないか
2. **依頼者の認識との整合**
   - 事実関係の記載のうち、依頼者本人でなければ真偽を確認できない部分をリストアップする
   - あわせて、依頼者への確認依頼文案（どの記載について何を確認してほしいか）を作成する
3. **依頼者が不安・不満を抱きそうな箇所**
   - 依頼者に不利な事実の記載、譲歩的な表現、認否の留保など、「なぜこう書くのか」と依頼者が疑問を持ちそうな箇所を特定する
   - 各箇所について、依頼者向けの説明文案（なぜこの書き方が有利または必要なのか）を作成する
4. **素人からの指摘への耐性**
   - 依頼者が言いそうな指摘（「もっと強く書いてほしい」「この事実も書いてほしい」「相手の言い分をなぜ認めるのか」等）を３〜５個想定する
   - それぞれについて、修正する場合は修正案を、修正しない場合はその理由（依頼者に説明できる形）を整理する

観点４の出力には、通常の指摘に加えて「依頼者向け確認事項リスト」（上記２で抽出したもの）を独立の項目として付す。
```
