## これまでの流れ
1. 発端の相談: 「スキルを共有フォルダに置き、フォルダにコピーガードをかけて流出防止できるか？」
   （参照された製品: コピーガードソフト https://www.tsslk.jp/reference/means/column_copysoft/ ※403で未取得）
2. 回答の要旨: コピーガード単体では限界がある。理由＝スキルは「読めなければ使えない」テキストで、
   画面撮影・書き写しといった抜け道（アナログホール）が残る。特にAIスキルは専用ビューア化すると
   肝心のAIから使えなくなる矛盾がある。ただし**営業秘密の「秘密管理性」を満たす証跡**としては価値あり。
3. 2案を提示し、ユーザーは **(B) ファイル配布せず、サーバー/リポジトリで一元管理（権限管理＋ログ）** を選択。
4. 提案書 `docs/skill-management-proposal.md` を作成・コミット・プッシュ済み。

## 提案の核心（「配らずに、使わせる」）
- スキルはサーバー（「スキルハブ」）上にのみ置き、使うたびに取得して**手元にファイルを残さない**。
- 毎回のサイクル: ①個人認証 → ②役割ベースの権限チェック → ③許可分だけ配信 → ④全アクセスをログ記録。
- 3つの柱: (1) 一元管理 (2) 役割ベース権限（退所時は1クリック遮断） (3) アクセスログ（抑止・異常検知・立証）。
- コピーガードより強い理由: 抜け道の前提（ファイルが手元にある状態）自体を無くせる。
- 法的連携: 不正競争防止法の営業秘密3要件、特に「秘密管理性」の裏付けになる。技術×規程（就業規則・秘密保持誓約）の両輪。

## ヒアリング済みの条件
- 事務所に **専任IT担当がいない** → 運用の手間が少ない・既製サービス活用を最優先。
- 設置形態（クラウド/オンプレ）は **未定**。→ 推奨は「まずクラウド既製(方式A)で小さく始め、高機密分だけ後からオンプレ切り出しのハイブリッド」。

## 次回（PCから）決めること = 未決事項
1. **スキルの種類**: AIに読ませるスキル / 人が読む業務ノウハウ / 両方 （←最重要・未確定。設計が変わる）
2. **設置形態の方針**: クラウド既製(方式A)で開始してよいか
3. **最初のスキル棚卸し**: PoC対象の代表スキルを3〜5個選ぶ
4. **想定規模**: 利用人数、機密度の分布

## 次にやると良いアクション
- 上記1〜4が決まれば、最小構成のPoC（認証・権限・ログの3点）設計に進む。
- 「スキルの棚卸し（公開可/所内限定/高機密のランク分け）」を最初にやるのが最重要（守る対象が決まらないと権限もログも設計できない）。
