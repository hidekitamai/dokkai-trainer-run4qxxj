# 事務所スキル一元管理システム 提案書

> 目的: 事務所のノウハウ（スキル）を **ファイルとして各人に配布せず**、サーバー／リポジトリで
> **一元管理・権限管理・アクセスログ** を行い、資産の流出を防ぎながら全所員が活用できる仕組みを作る。

前提条件（ヒアリング済み）:

- **専任IT担当がいない** → 運用の手間が少ないこと・既製サービス活用を最優先
- 設置形態（クラウド／オンプレ）は未定 → 両案を比較して提示
- 「スキル」の種類（AI用／人間用）は次回に確定 → 本書は両対応で設計

---

## 1. 設計思想：「配らずに、使わせる」

流出防止の要は、**手元にファイルを残さないこと**です。従来の「共有フォルダにファイルを置く」方式は、
コピー・持ち出しが構造的に可能でした。本提案は発想を逆転させます。

| | 従来（共有フォルダ方式） | 本提案（一元管理方式） |
|---|---|---|
| スキルの実体 | 各PCにコピーされる | **サーバー上にのみ存在** |
| 使うとき | ファイルを開く | **都度サーバーから取得（手元に残らない）** |
| 誰が使ったか | 分からない | **全アクセスをログ記録** |
| 権限 | フォルダ権限のみ | **人・役割ごとに細かく制御** |
| 退所時 | 手元のコピーが残る | **アクセス権を消せば即遮断** |

この「都度取得・手元に残さない」構造そのものが、コピーガードソフトより強力な流出防止になります。

---

## 2. 全体アーキテクチャ

中核は **「スキルハブ」= 事務所のスキルを配信する社内サーバー** です。

```
                       ┌─────────────────────────────────────────┐
                       │            スキルハブ (中核サーバー)          │
   所員 / AIクライアント │                                          │
   ┌──────────────┐    │  ①認証     ②権限チェック    ③スキル配信      │
   │ Claude 等のAI │───▶│  (誰か)  →  (使ってよいか) →  (該当分だけ返す) │
   │ or ブラウザ    │    │                     │                    │
   └──────────────┘    │                     ▼                    │
        ▲              │              ④アクセスログ記録              │
        │              │         (誰が・いつ・何を・どこから)          │
        └──────配信─────│                                          │
                       │   ┌────────────┐    ┌─────────────┐        │
                       │   │ スキル本体   │    │ 権限マスタ    │        │
                       │   │(サーバー内)  │    │(役割・個人)   │        │
                       │   └────────────┘    └─────────────┘        │
                       └─────────────────────────────────────────┘
```

処理の流れ（毎回このサイクル）:

1. **認証** — 所員本人かをSSO／トークンで確認（共有アカウント禁止、個人単位)
2. **権限チェック** — その人が、そのスキルを使ってよいかを判定（役割ベース）
3. **配信** — 許可された範囲のスキルだけを、その場限りで返す（ファイル保存はさせない）
4. **ログ記録** — 誰が・いつ・どのスキルを・どこから使ったかを全件記録

---

## 3. 権限管理（役割ベース：RBAC）

「フォルダに入れられるかどうか」ではなく、**役割（ロール）ごと**に許可を設計します。

| 役割 | 閲覧できるスキル例 | 管理操作 |
|---|---|---|
| 管理者（パートナー/事務局長） | 全スキル | 追加・更新・権限付与・ログ閲覧 |
| 弁護士（アソシエイト） | 担当分野のスキル一式 | 提案（下書き作成）まで |
| パラリーガル/事務 | 定型業務スキル | 閲覧・利用のみ |
| 外部協力者（一時） | 特定案件のスキルのみ・**期限付き** | 閲覧のみ、期間後は自動失効 |

ポイント:

- **個人単位で発行し、共有アカウントは作らない**（ログの意味を保つため）
- **退所・契約終了時は権限を1クリックで剥奪**（手元にファイルが無いので即座に遮断できる）
- 高機密スキルは「役割＋個別指名」の二重条件にできる

---

## 4. アクセスログ／監査

全アクセスを、後から追跡・立証できる形で記録します。記録項目の例:

| 項目 | 例 |
|---|---|
| 日時 | 2026-07-08 14:32:10 |
| 利用者 | 山田（弁護士・個人ID） |
| スキル名 | 「破産申立ドラフト作成」 |
| 操作 | 取得（read） |
| 接続元 | 事務所IP / VPN経由 |
| 結果 | 許可 / 拒否 |

活用:

- **抑止力**「利用は全部記録されている」と周知するだけで内部流出の抑止になる
- **異常検知** 短時間の大量アクセス（＝一括持ち出しの疑い）をアラート
- **立証** 万一の漏洩時、「誰が持ち出したか」「秘密として管理していたか」を証拠化

---

## 5. 流出防止の多層防御

技術単体では完璧になりません。次を重ねます。

1. **手元に残さない配信**（本方式の中核）
2. **個人単位の認証＋権限**（共有アカウント禁止）
3. **全アクセスのログ・監査**
4. **透かし／追跡**（配信内容に利用者IDを埋め込み、流出時に出所を特定）
5. **法的措置との連動**（→ 第8章。不正競争防止法上の「営業秘密」化）

---

## 6. 実現方式の比較（専任IT担当なし前提）

「スキル」がAI用か人間用かで最適解が変わりますが、いずれも**低運用負荷**を軸に選びます。

### 方式A：クラウドの既製サービスを組み合わせる（推奨・最短）

- 既存のプライベートリポジトリ（GitHub/GitLab等）や社内Wiki＋SSO＋アクセスログを活用
- **AIスキルの場合**: Claude等から接続する社内「スキル配信の窓口（MCPサーバー）」をマネージド環境に置く
- メリット: 構築・運用が軽い／専任IT不要／監査ログやSSOが標準装備
- 留意: データを国内リージョン・暗号化・アクセス制限で機密性を担保

### 方式B：オンプレミス（事務所内サーバー）

- スキル配信サーバーを事務所内に閉じて設置。外部にデータを一切出さない
- メリット: 機密性は最高
- 留意: サーバーの構築・保守・バックアップに手間 → **専任IT担当がいない現状ではハードルが高い**

### 推奨

> **まず方式A（クラウド既製サービス）で小さく始める。** 機密性の懸念が具体化した段階で、
> 機密度の高いスキルだけを方式B（オンプレ）へ切り出す **ハイブリッド** が現実的。

---

## 7. 段階的導入プラン

| フェーズ | 内容 | 目安 |
|---|---|---|
| **0. 棚卸し** | 「事務所のスキル」を洗い出し、機密度でランク分け（公開可／所内限定／高機密） | 1〜2週 |
| **1. PoC（小さく検証）** | 代表スキル3〜5個で、認証・権限・ログの最小構成を試す | 2〜4週 |
| **2. パイロット** | 1チームで実運用。使い勝手とログ運用を確認 | 1〜2ヶ月 |
| **3. 本番展開** | 全所展開＋就業規則・秘密保持誓約と連動 | 〜 |

「棚卸し」を先にやることが最重要です。**何を秘密として守るのかが決まらないと、権限もログも設計できません。**

---

## 8. 法的保護との連携（営業秘密化）

本システムは、技術対策であると同時に **不正競争防止法上の「営業秘密」3要件の裏付け** になります。

| 要件 | 本システムが果たす役割 |
|---|---|
| **秘密管理性** | アクセス制限・個人認証・ログが「秘密として管理していた」証跡になる ★ |
| 有用性 | スキル＝事務所ノウハウなので通常満たす |
| 非公知性 | 外部配布しない設計により維持 |

あわせて整備すべき規程:

- 就業規則・秘密保持誓約書に「事務所スキルは営業秘密」と明記
- 退所時のアクセス権剥奪・利用停止の手続き
- スキル利用ガイドライン（持ち出し・外部共有の禁止）

**技術×規程の両輪**で、初めて「守れる資産」になります。

## 9. 次のステップ（次回・PC からの検討事項）

次回、以下が決まれば具体設計・PoC構築に進めます。

2. **設置形態の方針** — クラウド既製（方式A）で開始してよいか

4. **想定規模** — 利用人数、機密度の分布

---

### 参考：本提案が「コピーガードソフト」より優れる理由（要約）

コピーガードは「ファイルは配るが持ち出しにくくする」対症療法で、画面撮影・書き写しといった
抜け道が残ります。本提案は **そもそもファイルを配らず、都度サーバーから取得して手元に残さない**
ため、抜け道の前提自体を無くし、かつ **誰が使ったかを完全に記録** できます。
